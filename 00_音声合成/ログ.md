# 対話ログ仕様書（評価基盤連携用）MVP版

**プロダクト:** FAQ＋エージェント絞り込み対話型チャットボット
**用途:** 対話評価機能（管理者運用画面）への連携データ仕様
**版数:** v1.0（MVP）
**作成日:** 2026年4月26日

---

## 1. 目的と前提

### 1.1 目的

本書は、対話評価機能のMVP（モニタリング＆改善、会話一覧、会話詳細、評価テスト実行、テスト設定、レポート出力 の6画面）が必要とするログ情報を、**対話システム側がどのような構造で送出すべきか**を定義する。**MVP版では必須項目のみを記載**し、推奨項目・任意項目は次フェーズで追加する。

### 1.2 設計方針

- **OpenTelemetry / OpenInference 準拠**: 業界標準のGenAI観測規格に沿う
- **1会話 = 1トレース**、**1処理ステップ = 1スパン** の階層構造で送出
- 評価機能側はトレース受信後に**スコアを書き戻す**（読み書き両方向）
- 個人情報は対話システム側でマスキング処理してから送出することを原則とする

### 1.3 用語

| 用語 | 説明 |
|---|---|
| トレース | 1会話セッション分の全ログの束（ルート単位） |
| スパン | トレース内の1処理ステップ（LLM呼出、検索など） |
| 属性 | スパンに付随するメタ情報のキー・バリュー |
| イベント | スパン内で発生した時刻付きの記録 |

---

## 2. 全体構造

```
Trace (1会話セッション)
│
├─ Span: turn (ターン1: ユーザー発話 → ボット応答)
│   ├─ Span: faq_retrieval        (FAQ検索)
│   ├─ Span: llm_call              (LLM呼び出し)
│   ├─ Span: tool_call             (ツール/MCP呼び出し)  ※ツール使用時のみ
│   └─ Span: decision              (意思決定: 即回答/絞り込み/エスカレ)
│
├─ Span: turn (ターン2)
│   └─ ...
│
└─ Event: session_end (終話イベント)
```

## 3. 必須ログ項目（MVP）

### 3.1 会話セッション（Trace属性）

| 項目 | 型 | 説明 |
|---|---|---|
| `trace_id` | string | 会話セッションの一意ID（UUID推奨） |
| `session_id` | string | 業務上の会話ID（trace_idと同じでも可） |
| `user_id` | string | ユーザーID（匿名IDで可） |
| `channel` | enum | Web / LINE / App / 電話 などのチャネル |
| `started_at` | timestamp | 会話開始日時（ISO 8601） |
| `ended_at` | timestamp | 会話終了日時 |

### 3.2 ターン（turnスパン）

| 項目 | 型 | 説明 |
|---|---|---|
| `span_id` | string | ターンの一意ID |
| `turn_number` | int | 何ターン目か（1始まり） |
| `start_time` | timestamp | ターン開始 |
| `end_time` | timestamp | ターン終了 |
| `user_input.text` | string | ユーザー発話テキスト（マスキング済） |
| `bot_output.text` | string | ボット応答テキスト |
| `bot_output.referenced_faq_ids` | array | 参照したFAQ IDのリスト |

### 3.3 FAQ検索（faq_retrievalスパン）

※ 検索方式（method）は自社のQA検索機能を利用するため、ログ項目には含めない。

| 項目 | 型 | 説明 |
|---|---|---|
| `query.raw` | string | 元のユーザー発話 |
| `candidates` | array of {faq_id, score, rank} | Top-K候補（K=5推奨） |
| `selected_faq_id` | string | 最終的に採用されたFAQ ID（採用なしの場合 null） |
| `selected_confidence` | float | 採用時の確信度（0.0〜1.0） |

### 3.4 LLM呼び出し（llm_callスパン）

| 項目 | 型 | 説明 |
|---|---|---|
| `model.name` | string | モデル名（例: gpt-x） |
| `model.version` | string | モデルバージョン |
| `prompt.template_id` | string | 使用したシステムプロンプトID |
| `prompt.template_version` | string | プロンプトバージョン |
| `prompt.input_messages` | array | 入力メッセージ配列（system/user/assistant） |
| `prompt.tokens_in` | int | 入力トークン数 |
| `completion.text` | string | 生成テキスト |
| `completion.tokens_out` | int | 出力トークン数 |
| `latency_ms` | int | 呼び出しレイテンシ |
| `purpose` | enum | answer / clarification / intent / end_check |

### 3.5 ツール呼び出し（tool_callスパン）※ツール使用時のみ

| 項目 | 型 | 説明 |
|---|---|---|
| `tool.name` | string | ツール名 |
| `arguments` | object (JSON) | 入力引数 |
| `result` | object (JSON) | 返り値（大きい場合は要約） |
| `latency_ms` | int | 実行時間 |

### 3.6 意思決定（decisionスパン）

| 項目 | 型 | 説明 |
|---|---|---|
| `decision.type` | enum | direct_answer / clarification / escalation / fallback |
| `decision.confidence` | float | 確信度 |

### 3.7 ユーザーフィードバック

| 項目 | 型 | 説明 |
|---|---|---|
| `feedback.thumbs` | enum | up / down / null |
| `feedback.csat_final` | int | 会話末尾の総合CSAT（1〜5） |

### 3.8 終話・結末情報（session_endイベント）

| 項目 | 型 | 説明 |
|---|---|---|
| `end.reason` | enum | resolved / escalated / user_timeout / user_abandoned / system_error |
| `end.containment_flag` | boolean | ボット完結フラグ（有人転送なし） |
| `end.resolution_flag` | boolean | 実解決フラグ（CSAT or 後追い判定） |
| `end.total_turns` | int | 総ターン数 |

### 3.9 システム・環境情報（Trace属性に追加）

| 項目 | 型 | 説明 |
|---|---|---|
| `env.environment` | enum | production / staging / development |

## 4. データ構造例（YAML）

```yaml
trace:
  trace_id: "01HXYZ123ABCDEF"
  session_id: "sess-20260426-0042"
  user_id: "u-1903"
  channel: "web"
  started_at: "2026-04-26T10:23:14+09:00"
  ended_at: "2026-04-26T10:31:00+09:00"
  env:
    environment: "production"

  spans:
    - span_id: "turn-1"
      kind: "turn"
      turn_number: 1
      start_time: "2026-04-26T10:23:14+09:00"
      end_time: "2026-04-26T10:23:16+09:00"
      user_input:
        text: "配送日変更したい"
      bot_output:
        text: "配送先住所のご変更も併せてご希望ですか？"
        referenced_faq_ids: []
      child_spans:
        - kind: "faq_retrieval"
          query:
            raw: "配送日変更したい"
          candidates:
            - { faq_id: "FAQ-203", score: 0.82, rank: 1 }
            - { faq_id: "FAQ-198", score: 0.71, rank: 2 }
            - { faq_id: "FAQ-211", score: 0.58, rank: 3 }
          selected_faq_id: null
          selected_confidence: 0.82
        - kind: "llm_call"
          model: { name: "gpt-x", version: "2026-04" }
          prompt:
            template_id: "main_v12"
            template_version: "v12.1"
            input_messages: [ { role: "system", content: "..." }, { role: "user", content: "配送日変更したい" } ]
            tokens_in: 312
          completion:
            text: "配送先住所のご変更も併せてご希望ですか？"
            tokens_out: 24
          latency_ms: 1200
          purpose: "clarification"
        - kind: "decision"
          decision:
            type: "clarification"
            confidence: 0.82

    - span_id: "turn-2"
      kind: "turn"
      # ... 省略

  events:
    - type: "user_feedback"
      attributes:
        thumbs: "down"
        csat_final: 3
    - type: "session_end"
      attributes:
        reason: "user_timeout"
        containment_flag: true
        resolution_flag: false
        total_turns: 5
```

## 5. 評価機能との対応関係（MVP）

評価機能側のどの画面がどのログ項目を必要とするか整理した対応表。

| 評価機能（MVP） | 必須となる主要ログ項目 |
|---|---|
| FE-01 モニタリング＆改善（KPI集計） | session_end イベント、feedback |
| FE-01 モニタリング＆改善（改善アクション） | user_input、selected_faq_id（null判定）、candidates、feedback、session_end（改善ロジックで集計） |
| FE-02 会話一覧 | Trace属性、session_end、feedback |
| FE-03 会話詳細（タイムライン） | 全ターンの user_input / bot_output |
| FE-03 会話詳細（FAQマッチング可視化） | faq_retrieval スパン全項目 |
| FE-03 会話詳細（スコア表示） | llm_call、decision（評価エンジン側で算出） |
| FE-04 評価テスト実行 | テスト実行は別経路（評価データセット直接実行）。本仕様外 |
| FE-05 テスト設定 | 本仕様外（管理画面内のCRUD） |
| FE-06 レポート出力 | 上記KPI集計結果のスナップショット |

## 6. プライバシー・PII取り扱い（運用ルール）

MVPではプライバシー関連のログ項目を持たず、運用ルールとして以下を遵守する。

**対話システム側で実施する**: 住所、電話番号、メールアドレス、クレジットカード番号、本人を特定可能な氏名等の自動マスキング。マスク済みのデータを評価基盤へ送出する。

**評価基盤側で実施する**: 表示時のロール別マスク、復号閲覧時の監査ログ記録。

※ マスキングルールのバージョン管理、同意区分のレコード単位での記録は次フェーズで対応する。

---

## 7. 送信プロトコル

- **送信方式**: OpenTelemetry Collector経由でOTLP（gRPC/HTTP）でリアルタイム送出
- **バッファリング**: スパン送出失敗時のローカルバッファ（30分以上）と再送機構を実装
- **サンプリング**: MVP期間は100%トレース送出（評価コスト調整は次フェーズ）

---

## 8. バリデーション

評価基盤側で受信時に以下を検証し、失敗時はDLQに退避して警告を出す。

- `trace_id` の一意性
- 必須項目の欠損チェック
- `started_at` ≤ `ended_at` のチェック
- ターン番号の連続性
- `referenced_faq_ids` 内のIDがKBに存在するかチェック（バッチで実行）

## 10. 次フェーズで追加予定の項目

MVP安定運用後、以下を順次追加することで評価の精緻化を進める。

**Phase 2 追加候補**

- システム・環境情報の拡充（`env.release_id`、`env.bot_version`、`env.faq_kb_version`）— 変更履歴とスコア変動の突合に必要
- プライバシー関連の項目（`privacy.masking_applied`、`privacy.masking_rule_id`、`privacy.consent_category`）— 監査対応の精緻化
- FAQ検索方式の記録（`method`）— 検索エンジン切替時の比較が必要になった場合
- 推奨項目の取り込み（faq_version、disclaimer_added、reranker_used、cost_usd 等）
- ツール呼び出しのエラー詳細、意思決定のpolicy・reason
- フィードバックの自由コメント・micro CSAT

**Phase 3 追加候補**

- 任意項目の取り込み（device、referrer、button_id 等）
- A/Bテスト群識別子（ab_test_arm）
- 詳細なリトライ・サンプリング情報
