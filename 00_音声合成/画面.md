# 対話評価画面 画面仕様書

**プロダクト:** FAQ＋エージェント絞り込み対話型チャットボット 管理者運用画面
**機能:** 対話評価（Conversation Evaluation）モジュール
**版数:** v1.0（MVP）
**作成日:** 2026年4月26日
**目的:** 他開発者が同等の機能を有したシステムを実装できる粒度で各画面を定義する

---

## 1. 全体構成

### 1.1 システム概要

本機能は、FAQ＋エージェント型チャットボットの**対話品質を継続的に可視化・改善する**ための管理者向け運用画面である。MVPでは以下4つの提供価値を実現する。

1. KPIを簡単に確認・共有できる
2. 簡単に改善点を確認できる
3. 簡単に改善ができる（自動改善・有人への改善提案）
4. 簡単に対話テストができる

### 1.2 画面一覧

| ID | 画面名 | 主な用途 |
|---|---|---|
| FE-01 | モニタリング＆改善（ダッシュボード） | KPI確認＋改善アクション実行を一画面で完結 |
| FE-02 | 会話一覧 | 本番運用中の会話の検索・絞り込み |
| FE-03 | 会話詳細 | 1会話の深掘り（タイムライン・スコア） |
| FE-04 | 評価テスト実行 | テスト起動・テスト結果詳細・テスト由来改善 |
| FE-05 | テスト設定 | 評価データセット ＋ AI評価器 の管理（2タブ） |
| FE-06 | レポート出力 | 月次／四半期レポートのエクスポート |

### 1.3 想定ロール

| ロール | 主な利用画面 | 業務 |
|---|---|---|
| サポートマネージャー | FE-01, FE-02, FE-03, FE-06 | KPI監視・改善承認・経営報告 |
| FAQ管理者 | FE-01, FE-03, FE-05 | FAQ起票・改訂、テストデータ整備 |
| AIチューニング担当 | FE-04, FE-05, FE-03 | プロンプト調整・評価器調整・テスト実行 |

## 2. 共通仕様

### 2.1 全体レイアウト

すべての画面で以下のレイアウトを共有する。

- 左: 固定幅のサイドバー（ナビゲーション）
- 右: トップバー（上部）＋ メインコンテンツ（スクロール可）

### 2.2 共通UIコンポーネント

| コンポーネント | 役割・特徴 |
|---|---|
| バッジ | 状態を識別表示。セマンティック5種（成功 / 要注意 / 失敗 / 情報 / 中立） |
| ボタン | 主要アクション（primary）／標準／サブ／小サイズ の4種類 |
| カード | コンテンツ単位のブロック。ヘッダー（左: タイトル/サブタイトル、右: アクション）＋本文 |
| KPIカード | 「ラベル ＋ 大きな数値 ＋ 前期比デルタ（改善 / 悪化 / 横ばい）」の3段構成 |
| テーブル | ヘッダー＋行で構成。行クリックで詳細遷移する場合あり |
| フィルタバー | 検索／セレクト／日付ピッカーを横一列、右端にアクションボタン |
| タブ | 同一カテゴリ内のビュー切替。アクティブタブは視覚的に区別 |
| アコーディオン／モーダル | フォームや詳細情報を必要時に表示 |

### 2.3 サイドバー仕様

- 固定幅のサイドバー（コンテンツを圧迫しない範囲）
- 上部にプロダクト名「対話評価機能」＋ サブ「対話品質の見える化と改善」
- 「運用」「評価」「設定」の3セクションでナビ項目をグループ化
- アクティブ項目は視覚的に強調
- ナビ項目は `data-screen` 属性で画面と1対1で紐付く

#### ナビ項目構成

| セクション | ナビ項目 | data-screen |
|---|---|---|
| 運用 | モニタリング＆改善 | `dashboard` |
| 運用 | 会話一覧 | `conversations` |
| 評価 | 評価テスト実行 | `jobs` |
| 評価 | テスト設定 | `test-config` |
| 設定 | レポート出力 | `reports` |

### 2.4 トップバー仕様

- 左: 画面タイトル（選択中の画面に応じて自動更新）
- 中央: 共通フィルタ（期間プリセット、チャネル）
- 右: ユーザーメニュー（アバター ＋ ユーザー名）

## 3. FE-01: モニタリング＆改善

### 3.1 目的

KPIモニタリングと改善アクション実行を1画面で完結させる、システムのトップ画面。

### 3.2 レイアウト構成（縦並び）

1. KPIカード（6枚、グリッド）
2. 主要指標の推移チャート（全幅）
3. 対話結末の内訳（ドーナツ）／ FAQ参照数Top10（横棒） の2列構成
4. 改善アクション
   - カテゴリ別サマリーカード4枚
   - フィルタバー
   - 改善提案リスト（全件）

### 3.3 ① KPIカード

| 項目 | 値の例 | デルタ表示 |
|---|---|---|
| 自己解決率 | 78.2% | 改善 +2.1pt（前週比） |
| 問題解決率 | 64.5% | 悪化 -0.8pt（前週比） |
| 顧客満足度 | 4.2 / 5 | 横ばい |
| 回答不能率 | 6.1% | 悪化 +0.4pt（前週比） |
| 有人回避率 | 71.8% | 改善 +1.5pt（前週比） |
| 平均対話往復数 | 3.4 | 改善 -0.2 |

定義:

- 自己解決率: 有人転送せず会話完結した割合
- 問題解決率: 実際に問題解決した会話の割合
- 顧客満足度: ユーザーCSAT 1-5の平均
- 回答不能率: 「分かりません」で終わった割合
- 有人回避率: 有人エスカレを回避できた率（Deflection）
- 平均対話往復数: 1会話あたりのターン数

### 3.4 ② 主要指標の推移

- 折れ線チャート
- カードヘッダーに指標選択ドロップダウン（自己解決率／問題解決率／顧客満足度／回答不能率）
- データ: 過去30日分の日次値（複数指標重ね表示可）

### 3.5 ③ 対話結末の内訳 / FAQ参照数Top10

2列等幅レイアウト。

- **対話結末**: ドーナツ。データ `[解決64%, 一部解決14%, 有人転送12%, 離脱10%]`、結末ごとに success/info/warning/danger を割り当て
- **FAQ参照数 Top10**: 横棒。10件、primary色

### 3.6 ④ 改善アクション

最も重要なセクション。検知シグナル＋改善案＋ワンクリック適用を統合表示する。

#### カテゴリ別サマリーカード（4枚）

| カテゴリ | 件数 | 影響件数表示 |
|---|---|---|
| 新規FAQの追加候補 | 3 | 影響: 約 173件 / 月 |
| 既存FAQの改訂候補 | 2 | 影響: 約 180件 / 月 |
| プロンプト調整提案 | 2 | 影響: 約 45件 / 月 |
| 検索チューニング | 1 | 影響: 約 53件 / 月 |

#### フィルタバー

フィルタ項目を横並びに配置し、右端に件数表示。

オプション:
- カテゴリ: すべて / FAQ追加 / FAQ改訂 / プロンプト調整 / 検索チューニング
- 影響度: すべて / 大（100件以上）/ 中（30-99件）/ 小（30件未満）
- 状態: すべて / 未対応 / レビュー中 / 承認済 / 却下

#### 改善アクションカード（リスト）

各カードは横一行で次の要素を含む。

- カテゴリを示す視覚的識別子
- 検知元シグナル（severityで識別表現を切替）
- 改善内容のタイトル
- カテゴリラベル、影響件数、想定工数
- アクションボタン（「詳細」と「承認して適用」または「レビュー依頼」）

##### 改善カードのデータ構造

```typescript
interface Improvement {
  id: string;
  cat: 'FAQ追加' | 'FAQ改訂' | 'プロンプト調整' | '検索チューニング';
  severity: 'high' | 'med' | 'low';
  signal: string;        // 検知元シグナル
  title: string;         // 改善内容のサマリ
  impact: number;        // 月間影響件数
  evidence: string;      // 詳細な根拠
  action: string;        // 推奨アクションの説明
  draft: string;         // 改善案のドラフト本文
  effortLevel: string;   // 想定工数（小5分・中30分・大1時間以上）
  expectedImpact: string; // 想定効果
  canAutoApply: boolean; // ワンクリック適用可否
}
```

##### severityによる表示の切替

検知シグナルのピルは severity により以下のセマンティックを示す。

- high: danger（緊急対応が必要）
- med: warning（要注意）
- low: info（参考情報）

##### アクションボタンの切替

- `canAutoApply: true` → 「承認して適用」（primaryボタン）
- `canAutoApply: false` → 「レビュー依頼」（primaryボタン）

### 3.7 サンプル改善データ（MVP想定の7件）

```
imp-001: FAQ追加 / high / 「領収書を再発行したい」76件 / 自動適用可
imp-002: FAQ追加 / med  / 「ギフト包装は可能？」64件 / 自動適用可
imp-003: FAQ改訂 / high / FAQ-088 表現バリエーション不足 142件 / 自動適用可
imp-004: プロンプト調整 / high / 終話前確認ターン追加 42件 / レビュー必須
imp-005: プロンプト調整 / high / グラウンディング強化 3件 / レビュー必須
imp-006: FAQ改訂 / med  / FAQ-203/198 混同解消 38件 / 自動適用可
imp-007: 検索チューニング / low / 同義語辞書追加 53件 / 自動適用可
```

## 4. FE-02: 会話一覧

### 4.1 目的

本番運用中の全会話を検索・絞り込みし、評価対象を素早く特定する。

### 4.2 フィルタバー

横並びで以下を配置し、右端に「CSV出力」「一括で評価データに追加」ボタン。

- 検索: ユーザーID / 会話ID / FAQ ID / キーワードの自由入力
- 期間プリセット: 過去30日（既定）/ 本日 / 過去7日 / 過去30日 / 過去90日 / カスタム
- 開始日・終了日の2つの日付ピッカー
- 結末: すべて / 解決 / 有人転送 / 離脱
- 満足度: すべて / 高（4〜5）/ 中（3）/ 低（1〜2）
- FAQ分野: すべて / 配送 / 返品 / 支払い / 会員情報

### 4.3 テーブル列

| カラム | 型 | 説明 |
|---|---|---|
| チェックボックス | checkbox | 一括操作用 |
| 会話ID | string | クリックで詳細遷移 |
| 開始日時 | datetime | YYYY/MM/DD HH:MM |
| ユーザー | string | 匿名ID |
| 最終FAQ | string | 参照したFAQ ID＋タイトル |
| 結末 | badge | 解決(ok)/有人転送(warn)/離脱(ng)/タイムアウト(ng) |
| 満足度 | badge | 数値＋色（4-5=ok / 3=warn / 1-2=ng） |
| 総合スコア | badge | 0-1の値＋色（≥0.85=ok / ≥0.7=warn / <0.7=ng） |

### 4.4 ページネーション

- ボトムに件数表示「1〜20件 / 全 1,283件」
- ページボタン: ‹ 1 2 3 4 5 ›

### 4.5 データ構造

```typescript
interface Conversation {
  id: string;           // 例: 'c-20260426-0042'
  date: string;         // 例: '2026/04/26 10:23'
  user: string;         // 例: 'u-1903'
  faq: string;          // 例: 'FAQ-203 配送日変更' or '(該当なし)'
  turns: number;        // 内部値（表示しない）
  outcome: '解決' | '有人転送' | '離脱' | 'タイムアウト';
  csat: number | null;  // 1-5 or null
  score: number;        // 0.0-1.0
  status: 'unreviewed' | 'reviewing' | 'reviewed' | 'badContain';
}
```

## 5. FE-03: 会話詳細

### 5.1 目的

1セッションをタイムライン・推論軌跡・FAQマッチング遷移・評価スコアの4軸で立体的に確認し、改善アクションへ即移行する。

### 5.2 レイアウト

```
[← 会話一覧へ戻る]
縦並びの構成:

1. 上部: 「会話一覧へ戻る」リンク
2. ヘッダー: 会話ID / ユーザー（右上に「Bad Containment候補」のラベル）
3. メタ情報グリッド（開始 / 終了 / 往復数 / CSAT / 最終FAQ / 自己解決判定 / 解決状態 / チャネル）
4. 本体（2列構成）:
   - 左: タイムライン（ターン毎の発話、トレース行、FAQ Top-K候補表示）
   - 右: 評価スコア（ターン単位 / 対話単位 / 軌跡単位 / ユーザー評価 / 改善ポイント）
5. アクションバー（ゴールデン追加 / レビュー依頼 / FAQ起票エスカレ など）
```

### 5.3 メタ情報グリッド

| 項目 | 値の例 |
|---|---|
| 開始 | 2026/04/26 10:23 |
| 終了 | 2026/04/26 10:31 |
| 往復数 | 5 |
| 満足度 | 3 / 5 |
| 最終FAQ | FAQ-203 配送日変更 |
| 自己解決判定 | 該当（要確認） |
| 解決状態 | 未解決（強調表示） |
| チャネル | Web |

### 5.4 タイムライン

ターン毎に以下を表示。

#### ユーザー発話 / ボット応答

- 発話者を識別する小さな丸型アバター（ユーザーとボットで色を分ける）
- ヘッダー: 「T{n} 発話者 HH:MM:SS」
- 吹き出し型のテキストブロック

#### トレース行（薄背景の補足ブロック）

- **FAQ検索**: `候補上位3件（ベクトル＋キーワード併用検索, 240ms）：FAQ-203(0.82) FAQ-198(0.71) FAQ-211(0.58)`
- **AI応答生成**: `gpt-x · 1.2秒 · 312トークン · 動作: 意図が曖昧なため逆質問`
- **会話終了**: `理由: ユーザー応答なし / 自己解決判定: 該当 / 実際の解決: 不成立`

### 5.5 評価スコアパネル

3つのカテゴリのカードを縦並びで表示。各スコアはラベル + 横バー + 数値で構成。

#### 1ターン単位の評価

| 項目 | 例 |
|---|---|
| 回答忠実度（FAQ準拠） | 0.95 |
| 回答関連性 | 0.88 |
| 参照FAQの適合度 | 0.82 |

#### 対話全体の評価

| 項目 | 例 |
|---|---|
| 対話完結度 | 0.55（warn） |
| 文脈保持度 | 0.92 |
| 絞り込み効率 | 0.70 |

#### AIの動作経路の評価

| 項目 | 例 |
|---|---|
| ツール選択の正確性 | 1.00 |
| 想定経路との一致度 | 0.90 |

#### ユーザー評価

- 「役に立たなかった」もしくは「役に立った」のシグナル表示
- 自由コメント（あれば）

#### この会話から検出された改善ポイント

- 視覚的に区別されたカード
- ヘッダー: 「この会話から検出された改善ポイント」
- 内容: 原因推定 + 推奨アクション
- ボタン: 「改善アクション一覧で対応する」（ダッシュボードへ遷移）

### 5.6 アクションバー

ボタンを横並びで配置:
- 評価データに追加
- 人手レビューを依頼
- FAQ追加・改訂を起票
- 操作履歴を表示（補助）

## 6. FE-04: 評価テスト実行

### 6.1 目的

評価データセット＋AI評価器でテストを実行し、結果を即座に確認し、テスト由来の改善提案にアクセスする。テスト系の動線は本番系と独立して完結する。

### 6.2 レイアウト構成

```
縦並びの構成:

1. テスト実行CTAカード（最上部に視覚的に強調して配置）
2. 過去のテスト一覧（テーブル＋再実行ボタン）
3. 最新テスト結果の詳細（4枚のサマリーKPI、テスト結果ケース一覧、改善提案）
4. 直近の評価テスト総合スコア推移（折れ線）
```

### 6.3 ① テスト実行CTAカード

- 背景: primary色から少し明るいprimary色への斜め方向グラデーション、文字色 白
- 構成:
  - 左: タイトル「対話テストを実行」＋ サブ「評価データセットと評価器を選択して、対話テストをワンクリックで起動できます」
  - 右:
    - ラベル「評価データセット」＋ select（本番回帰用_v3 / 配送カテゴリ抜粋 / 難ケース集 / 個人情報・コンプラ）
    - ラベル「評価器」＋ select（標準セット推奨 / FAQ参照型 / 個人情報チェック / 評価器v4）
    - 「テストを実行」プライマリボタン

### 6.4 ② 過去のテスト一覧

| 列 | 説明 |
|---|---|
| テストID | 例: `test-2042` |
| 名称 | テスト名 |
| 評価データセット | 例: `本番回帰用_v3` |
| 対象AIモデル | 例: `gpt-x@2026-04` |
| 評価器 | 例: `標準セット` |
| 状態 | バッジ: 完了(ok)/実行中(info)/失敗(ng) |
| 総合スコア | 数値 or `—` |
| 実行日時 | datetime |
| アクション | 「再実行」ボタン（実行中は disabled） |

### 6.5 ③ 最新テスト結果の詳細

#### サマリーKPI（4枚）

| 項目 | 例 | デルタ |
|---|---|---|
| 総合スコア | 0.89 | 改善 +0.01（前回比） |
| 合格ケース | 108 / 124 | 合格率 87.1% |
| 要確認 | 11 | スコア 0.6〜0.85 |
| 失敗ケース | 5 | スコア 0.6未満 |

#### テスト結果ケース一覧

タブ切替式（`data-test-tab`）。

| タブ | 件数 | スコア閾値 |
|---|---|---|
| 全件 | 124 | - |
| 合格 | 108 | ≥ 0.85 |
| 要確認 | 11 | 0.6〜0.85 |
| 失敗 | 5 | < 0.6 |

##### テーブル列

| 列 | 説明 |
|---|---|
| ケースID | 例: `case-087` |
| カテゴリ | 例: `配送日変更` |
| 想定質問 | 例: `「明日の配達を別の日に」` |
| 期待FAQ | 例: `FAQ-203` |
| 実際の挙動 | 例: `FAQ-198を誤マッチ` |
| スコア | バッジ（色分け） |
| 判定 | バッジ（合格/要確認/失敗） |

##### データ構造

```typescript
interface TestCase {
  id: string;       // 'case-087'
  cat: string;      // '配送日変更'
  q: string;        // 想定質問
  expected: string; // 'FAQ-203'
  actual: string;   // 実際の挙動説明
  score: number;    // 0-1
  status: 'passed' | 'warning' | 'failed';
}
```

##### ページネーション

全件タブと合格タブにはページネーションを表示。

#### このテスト結果からの改善提案

## 10. 共通動作仕様

### 10.1 画面遷移

`data-screen` 属性のクリックで `switchScreen(name)` を呼び出し、screen要素の `.is-active` を切替。

```javascript
function switchScreen(name) {
  document.querySelectorAll('.screen').forEach(s => s.classList.remove('is-active'));
  document.getElementById('screen-' + name)?.classList.add('is-active');
  // ナビ項目のアクティブ切替（detail画面は conversations を維持）
  const navName = name === 'detail' ? 'conversations' : name;
  document.querySelectorAll('.nav-item').forEach(n => n.classList.remove('is-active'));
  document.querySelectorAll(`.nav-item[data-screen="${navName}"]`).forEach(n => n.classList.add('is-active'));
  // タイトル更新
  document.getElementById('topbarTitle').textContent = titles[name];
  // チャート遅延初期化
  if (['dashboard', 'jobs'].includes(name)) setTimeout(initCharts, 50);
  // スクロールリセット
  document.getElementById('content').scrollTop = 0;
}
```

```javascript
window.toggleForm = function(id) {
  const el = document.getElementById(id);
  if (!el) return;
  el.style.display = el.style.display === 'none' ? 'block' : 'none';
  if (el.style.display === 'block') {
    el.scrollIntoView({ behavior: 'smooth', block: 'nearest' });
  }
};
```
